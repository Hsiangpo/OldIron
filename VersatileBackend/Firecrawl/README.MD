# Firecrawl 模块 (VersatileBackend)

## 简介

`Firecrawl` 模块是本通用后端中用来负责网页抓取、结构化抽取以及智能内容分析的核心组件。该模块高度封装了对 Firecrawl API 的调用，并衍生出多个维度的解析链路，以便各国家节点根据站点特性灵活拼装组合。

## 目录与子模块结构

本模块通过组合不同引擎（LLM 或 Rule）来实现对关键字段（如代表人、邮箱）的精准提取。各子模块的职责如下：

### 1. `src`

**Firecrawl 爬取主框架**

- 负责维护与 Firecrawl 的网络连接、参数配置（如页面范围、深度控制）以及容错重试。
- 它充当下游解析模块的数据“进货管道”，拉取回来的结构化 Markdown 或 HTML 会流转给下面具体的业务模块。

### 2. `Director`

**代表人/法人/高管提取专用件**

- 专注解决“这个站点的哪个字段是董事长/CEO”这类的提取工作。
- 可以根据复杂度，自由组合调用底层的 `LLM` 模块去理解复杂的文本段落，或者调用 `Rule` 模块直接拆解固定的表格/名片格式。

### 3. `Email`

**邮箱寻址与清洗专用件**

- 专注解决挖掘页面或联系我们 (`/contact`) 里的邮箱信息。
- 提取规则十分严谨，需负责剔除垃圾邮箱、平台占位符及无效信息。

### 4. `LLM`

**外部大模型协同件**

- 当 Firecrawl 返回的文本结构过于混乱、或者存在长篇大论的公司介绍信时，此时传统规则束手无策。
- 本模块会组装 prompt 并调用外部大模型（配置在全局 `.env` 中），使得 Firecrawl 拿回来的文本经过 LLM 智能理解后准确输出目标字段结构。

### 5. `Rule`

**轻量级启发式/规则基处理件 (Non-LLM)**

> **注：关于 Rule (规则) 的具体含义（参考日本站点的实现经验）：**
> "Rule" 主要是指不依赖 LLM，纯靠代码层面的正则表达式 (Regex)、启发式规则 (Heuristics) 以及黑白名单列表来提纯信息的机制。
> 例如参考日本 `site_agent` 的实现，Rule 主要涉及两方面：
>
> 1. **抽取提取（Heuristics）**：通过规则直接精准匹配网页源码里的结构化信息块（如 `JSON-LD` 数据、`<meta property="og:site_name">` 等标签），直接拿现成的结果。
> 2. **过滤清洗（Black/White lists）**：在提取邮箱时，通过名单过滤掉免费公共邮箱池（如 `qq.com`, `gmail.com`）、剔除无效占位符（如 `noreply`, `privacy`, `abuse`）以及用正则判断域名主从关系。
>
> 使用 Rule 模块的好处是在版式固定或标准元素明确时，极大地节省了调用 LLM 的延迟与成本。

## 融会贯通：模块协同策略

这四个处理子组件 (`Director`, `Email`, `LLM`, `Rule`) 并不是割裂的，而是可以在 `src` （主框架）调度下互相搭配使用：

1. **Firecrawl + LLM 组合**：
   - _场景_：针对版式千奇百怪、结构极其不规范的中小企业官网，需要抓取邮箱或代表人。
   - _特点_：最宽容，但也最费时间/token。
2. **Firecrawl + Rule 组合**：
   - _场景_：针对格式严谨、包含结构化标记（如大型黄页或官方登记站），需要抓取邮箱或代表人。
   - _特点_：速度最快，正则匹配和黑名单直接过滤，0 token 消耗。

3. **按字段拆分策略组合**：
   - 比如：由于代表人信息经常隐藏在复杂的 "关于我们" 纯文本里，可以采用 `Firecrawl + LLM` 来精准抽取 **代表人 (`Director`)**；同时，因为邮箱的格式（`@xxx.com`）具备极强正则特征，可以直接采用 `Firecrawl + Rule` 来轻量级获取 **邮箱 (`Email`)**。
